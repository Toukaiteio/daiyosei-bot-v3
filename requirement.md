# Daiyosei Bot V3 目标需求说明

## 1. 项目定位

本项目将从旧版 `daiyosei-bot-v2` 的遗留实现中彻底清理出来，重建为一个基于 `Node.js` 的现代化机器人系统。

新系统只保留 `OneBot` 协议支持，不再保留 Discord 平台适配。

系统核心不再内置复杂的自研大模型调用体系，而是基于 `OpenAI Agent SDK` 构建 Agent 层，再通过统一的 OpenAI 兼容模型接口完成推理、工具调用与任务编排。

长期记忆不再使用自研记忆库，统一切换为 `OpenMemory`。

## 2. 核心目标

1. 机器人不再是单纯的聊天陪伴型角色，而是以“解决问题的强大机器人”为主。
2. 陪伴聊天能力必须保留，但它是次级能力，不是唯一定位。
3. 机器人要具备明确的人设、稳定的工具使用能力、可配置的模型能力，以及跨群的连续记忆。
4. 整个系统必须支持现代化 WebUI 配置，不再依赖修改代码或静态配置文件完成主要设置。
5. 所有高风险能力都必须进入安全沙箱，避免模型直接执行危险操作。

## 3. 人设设计

### 3.1 基底人设

人设基底采用东方 Project 的「琪露诺」。

但这个人设不是单纯的可爱陪聊型设定，而是：

- 具备琪露诺的基础语言风格与性格特征
- 允许保留轻松、俏皮、带一点胡闹感的表达
- 同时具备强工具能力、问题解决能力和 Agent 推理能力
- 在需要时优先给出可执行方案，而不是只做情绪陪伴

### 3.2 角色策略

默认风格应偏向：

- 先解决问题
- 再进行补充说明
- 需要时再切换到陪伴、吐槽、轻松互动模式

也就是说，琪露诺的人设是“外壳”，解决问题的能力是“核心”。

## 4. 模型策略

系统中的模型配置需要支持分组管理。

### 4.1 模型分组

至少支持以下模型角色：

- 主模型：用于日常聊天、默认回复、基础任务处理
- 搜索模型：用于信息检索、搜索结果总结、联网知识整理
- 识图模型：用于图片理解、OCR、视觉任务
- 高参模型：用于复杂任务、Agent 内部深度推理、长链路决策

### 4.2 单模型模式

如果用户只配置了一个主模型：

- 所有任务默认交给主模型
- 如果主模型具备识图能力，则直接启用识图功能
- 如果主模型不具备识图能力，则启用图片转译兼容层
- 搜索任务和高参任务也回退给主模型

### 4.3 OpenAI 兼容要求

所有可配置模型都必须按 `OpenAI Compatible` 的方式接入。

模型配置中需要支持：

- `base_url`
- `api_key`
- `model`
- `supports_vision`
- `supports_reasoning`
- `reasoning_effort`

其中 `reasoning_effort` 需要支持预设推理力度，至少包括：

- `low`
- `medium`
- `high`

### 4.4 推理模型策略

对于推理模型：

- 可以在 WebUI 中显式标记其为推理模型
- 可以预设推理力度
- Agent 在复杂任务下优先调用高参模型
- 若未配置高参模型，则自动回退到主模型

## 5. Agent 与沙箱

系统基于 `OpenAI Agent SDK` 构建 Agent 能力。

### 5.1 Agent 职责

Agent 层负责：

- 规划任务
- 调用工具
- 分工执行
- 汇总结果
- 决定是否需要查看图片
- 决定是否需要搜索
- 决定是否需要调用高参模型

### 5.2 沙箱要求

Agent 不得直接拥有无限制执行权限。

所有脚本执行、系统命令执行、文件操作都必须经过安全沙箱。

沙箱需要至少做到：

- 默认禁止删除文件
- 默认禁止写入持久化敏感路径
- 默认禁止任意覆盖配置文件
- 默认禁止高风险系统命令
- 默认限制网络、文件系统和进程能力
- 对危险操作进行拦截、告警或显式确认

### 5.3 危险命令拦截

以下操作必须被认为是高风险行为：

- 删除文件或目录
- 覆盖持久化配置
- 批量修改历史数据
- 读写 QQ 账号敏感配置
- 执行未授权脚本

这些行为必须在沙箱层被拦截，而不是交给模型自由决定。

## 6. 多模态与图片机制

### 6.1 图片转译兼容层

必须保留图片转译兼容层，用于兼容不具备视觉能力的模型。

这个兼容层的职责是：

- 接收图片消息
- 生成结构化的图片转译结果
- 将图片内容转换为可被文本模型理解的上下文
- 为后续检索和追溯保存缓存

### 6.2 直接使用多模态模型

如果用户为当前场景直接配置了具备视觉能力的多模态模型作为默认模型：

- 可以跳过图片转译兼容层
- 直接由默认模型处理图片

也就是说，图片转译层是兼容方案，不是强制绕一层的固定流程。

### 6.3 图片解析缓存

图片解析结果必须缓存。

缓存内容至少包括：

- 图片哈希
- 来源消息 ID
- 群 ID
- 发言者 ID
- 发送时间
- 图片解析文本
- OCR 结果
- 标签
- 是否已被多模态模型直接解析

缓存的作用是：

- 避免重复识图
- 支持历史图片追溯
- 支持按时间、发言者、回复链找到对应图片

### 6.4 群聊注意力机制

群聊机器人不能“看到图片就自动看图”。

系统必须有一个注意力决策层，至少考虑：

- 当前消息是否明确要求看图
- 当前对话主题是否依赖该图片
- 该图片是否来自当前消息链
- 是否能从聊天记录中按时间、发言者、回复关系找到关联图片
- 是否已有缓存的图片解析结果可以直接复用

只有当 Agent 判断“看图是必要的”时，才调用图片解析工具。

### 6.5 图像追溯能力

Agent 需要能够基于聊天历史追溯图片，例如：

- 根据时间范围找对应图片
- 根据发言者追溯其发送过的图片
- 根据回复链找被引用的图片
- 根据上下文关键词定位历史图片

这意味着图片解析不仅是一次性动作，而是可检索、可回忆、可复用的上下文资产。

## 7. 记忆体系

### 7.1 跨群记忆

AI 必须拥有“对单个用户的跨群聊记忆”。

要求如下：

- 记忆必须按用户维度跨群共享
- 同一个用户在不同群中的重要信息应能被联想到
- 群内上下文和用户全局记忆需要分层处理

### 7.2 记忆层次

建议分为三层：

- 会话短期记忆：当前群聊上下文
- 用户长期记忆：跨群共享的用户信息
- 事件记忆：重要事件、关系变化、偏好变化

### 7.3 OpenMemory 接入

OpenMemory 是唯一的长期记忆系统来源。

系统需要通过适配层与 OpenMemory 对接，保证：

- 记忆写入可控
- 记忆检索可控
- 记忆更新可追踪
- 记忆删除和隐私控制可管理

## 8. WebUI 要求

必须提供现代化 WebUI，用于替代手工改配置。

### 8.1 必须可配置的内容

WebUI 至少需要支持：

- 模型配置
- 人设配置
- 插件管理
- skill 导入
- QQ 账号管理
- 记忆状态查看
- 日志和运行状态查看

### 8.2 QQ 账号管理

QQ 账号管理应覆盖：

- OneBot 连接状态
- 多账号管理
- 账号启停
- 心跳状态
- 黑白名单
- 账号级别的基础信息展示

### 8.3 插件与 skill 管理

WebUI 需要支持：

- 插件启用 / 禁用
- skill 导入
- skill 版本管理
- skill 权限说明
- skill 的沙箱级别配置

技能导入应以可声明的包格式为主，不依赖硬编码注册。

## 9. 协议与平台边界

### 9.1 保留项

- `OneBot` 协议
- QQ 机器人主场景
- 群聊与私聊能力
- 图片、文本、多模态输入

### 9.2 移除项

- Discord 平台支持
- 旧版 Python 运行时架构
- 自研记忆库
- 旧的硬编码模型请求链路
- 将业务逻辑绑死在单个模型供应商上的实现

## 10. 架构原则

1. 代码必须模块化，核心逻辑与接入层分离。
2. 模型能力通过配置描述，不通过硬编码分支描述。
3. 多模态能力必须可插拔。
4. 记忆能力必须外置到 OpenMemory。
5. 安全能力必须前置，不依赖“模型自觉”。
6. WebUI 是主要配置入口，而不是附属功能。
7. 所有旧架构内容都应视为迁移素材，不应继续作为新架构的运行基础。

## 11. 迁移验收标准

当以下条件满足时，迁移才算完成：

- 新项目运行于 `Node.js`
- 仅保留 `OneBot`
- `OpenAI Agent SDK` 已接管 Agent 层
- `OpenMemory` 已接管长期记忆
- WebUI 可完整配置模型、人设、账号、插件和 skill
- 图片转译兼容层和多模态直连模式都可用
- 群聊注意力机制正常工作，不会无条件见图就看图
- 用户跨群记忆可生效
- 危险脚本和危险文件操作可被沙箱拦截

## 12. 实施建议

建议将新工程拆分为以下能力域：

- `bot-core`
- `onebot-adapter`
- `agent-runtime`
- `memory-adapter`
- `vision-gateway`
- `plugin-system`
- `skill-loader`
- `web-console`
- `sandbox-executor`

旧版 Python 代码、Discord 适配、旧记忆实现、旧模型请求逻辑应在迁移过程中逐步移出主分支运行路径。
